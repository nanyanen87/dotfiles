# Task
**ユーザーとのやりとりは全て日本語を使うこと**
以下のセクションの番号順に作業を進めてください。
セクションの番号順に作業を進める上でイレギュラーな事象が発生した場合は、
ユーザーにその状況を伝え、どのように対応するか質問してください。

## 1. コンテキスト収集
- branch_current: !`git branch --show-current`
- status: !`git status --porcelain`
- args_raw: "$ARGUMENTS"

## 2. 参照Issueの取得
`conventional commits` のフッターに付与する `Refs` の値として、参照Issueを取得します。
このタスクの流れは以下の通りです。
1. `/commit` コマンドの引数で指定があれば、その値を使用。(例：`#123`)
2. `/commit` コマンドの引数がない場合、現在参照しているブランチから値を取得(例：`issues/123`→`#123`)
3. 1, 2 から値が取得できなければ `Refs: none` を設定

上記の値を以下のコンテキストに保存します。
- refs_resolved: !`bash -lc '
  trim(){ awk '\''{$1=$1}1'\''; }; 
  ARGS=$(printf "%s" "$ARGUMENTS" | trim);
  BR=$(git branch --show-current);
  if [ -n "$ARGS" ]; then
    echo "$ARGS";
  elif [[ "$BR" =~ ^issues/([0-9]+)([-_/].*)?$ ]]; then
    echo "#${BASH_REMATCH[1]}";
  else
    echo "none";
  fi
'`

この時、`{{ refs_resolved }}` が `none` もしくは不適切な形式の場合のみ、ユーザーに以下の注意喚起を行い、このまま進めて良いか確認します。
「参照Issueの値 (Refs) が `{{ refs_resolved }}` になっています。このままコミット作業に進んでよろしいですか？ (Yes/No)」

- Yes の場合：次のステップに進んでください。
- No の場合：`{{ refs_resolved }}` に設定する値をユーザーに質問し、得た回答を `{{ refs_resolved }}` にセットしてください。

### 注意点
- Refs値は現在の作業がGitHub上のどのIssueにリンクしているかを判断するために重要です。
- 適切なRefsの形式: `Refs: #{issue番号}`
- `{{ refs_resolved }}` が `#` で始まる数値形式（例：`#118`）の場合は、適切な値として次のステップに直接進んでください。

## 3. 変更ファイルのステージング
変更されたファイルをステージングエリアに追加します。

1. `{{ status }}` の内容をユーザーに分かりやすく表示してください。
2. 表示されたすべてのファイルをステージングするかユーザーに確認してください。(Yes/No)
3. Yesの場合： `git add -A` を実行してください。
4. Noの場合： ユーザーに手動でのステージングを案内してください。

### 注意点
- `{{ status }}` の内容が100ファイルを超える場合は、全件表示ではなく要約して表示してください。
- `git add -A` の実行により問題が生じる可能性があるファイルが含まれている場合は、事前に注意を喚起してください。
- 質問時にclaude codeへステージングしたいファイルを伝えればそれらを`git add`できることも伝えてください。
- 質問時にNoを選択した場合の動作（手動ステージングを待つこと）を事前に説明してください。


## 4. コミット前コード品質チェック
コミットメッセージ生成前に以下の品質チェックを実行し、**今回変更したファイルに関連するエラーのみ**を対象として問題があれば修正を促してください：

1. **型チェック**: `npm run typecheck` - 変更ファイルに関連するTypeScriptエラーがあれば修正を促す
2. **リント**: `npm run lint` - 変更ファイルに関連するESLintエラーがあれば修正を促す  
3. **フォーマット**: `npm run format:check` - 変更ファイルに関連するフォーマットエラーがあれば `npm run format` の実行を促す

### 注意点
- 今回の作業範囲外（既存のエラー）は無視して構いません。git diffやgit statusで確認できる変更ファイルに関連するエラーのみを対象としてください。

変更ファイルに関連する品質チェックが通過した場合のみ、コミットメッセージの生成に進んでください。

## 5. コミットメッセージの作成
以下の手順でConventional Commits形式のコミットメッセージを作成します：

1. **変更内容の確認**: `git diff --cached` でステージされた変更内容を詳細に確認
2. **コミットメッセージ生成**: 変更内容に基づいて以下の **厳格なルール**でコミットメッセージ案をコードブロックで出力（コミットは実行しない）

### 形式（必須）
- ヘッダ: `type(scope?): description`
  - `type`: `feat|fix|docs|style|refactor|perf|test|build|ci|chore|revert` から適切に。
  - **日本語主体**で、必要なら英語の方が自然な専門用語は英語のまま。
  - Emoji Prefixを適切に使用する。使用するのは description の先頭。
  - 破壊的変更は `type(scope)!: ...` もしくはフッター `BREAKING CHANGE: ...` を使用。
- 本文: **日本語主体**で、必要なら英語の方が自然な専門用語は英語のまま。目的・背景・影響・代替策を簡潔に（72字前後で折り返しやすく）。
- フッター（必須）:
  - `Refs: ${refs_resolved}`
  - 必要に応じて `BREAKING CHANGE:` を追加（影響範囲と移行手順を具体的に）。

### 注意点
- コンテキストから、今回編集したファイルのみを対象にすること。
- コンテキストがない場合、コミット対象のファイルを質問すること。

## 6. コミットの実行
`## 5`で作成したコミットメッセージを使用してコミットの実行を行います。

1. ユーザーに、`## 5`で表示した内容でコミットを実行して良いか確認します（Yes/No）
2. Yesならコミットを実行します。
3. Noなら、ユーザーと対話してコミットメッセージを確定します。

### 注意点
- husky による `pre-commit` フックが実行されます。  
  `## 4`のコミット前コード品質チェックで適切な対応を行っていない場合、エラーが起きてコミットできない場合があります。
  その場合は、エラーの内容をユーザーに伝え、対応を促してください。

## 7. リモートリポジトリへのプッシュ
`## 6`でコミットした内容をリモートリポジトリへプッシュします。

1. ユーザーに、コミットした内容をリモートリポジトリへプッシュするか確認してください（Yes/No）
2. Yesの場合、リモートリポジトリへプッシュを実行してください。
3. Noの場合、タスクを終了してください。

### 注意点
- プッシュ時は、どのブランチにプッシュするか明記してください。
- `development`・`main`ブランチの場合は、他の開発者への影響範囲を確認してから実行するよう注意喚起してください。
